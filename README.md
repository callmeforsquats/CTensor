# Тензорная библиотека на чистом C
## Предыстория
Я увлёкся глубоким машинным обучением, поэтому регулярно изучал pytorch и делал на нём всякое.
Однако мне всё время было интересно, как pytorch устроен изнутри, что скрывается за этими model.zero_grad(), optimizer.step() и тд.
Поэтому я захотел создать нечто подобное, свою тензорную микробиблиотеку самостоятельно, с полного нуля.
Несмотря на то, что pytorch реализован на C++, я осознанно выбрал чистый C. 
Этот выбор был, в основном, идеалогический: мне хотелось реализовывать всё с полного нуля, на чистых malloc/free, а не учить семейство указателей, исключений, stl и тд. Я, конечно, понимал, что можно использовать C++ как C с классами, но всё же решил отказаться от возможного соблазна использовать абстракции типа STL.

P.S. Теперь даже C++ кажется мне чем-то высокоуровневым.

## Стек
- чистый C
- cmake
- nvcc (подразумевалась cuda, поэтому процесс сборки строился вокруг nvcc)
- clang-cl (потому что MSVC мягко говоря консервативный, а про WSL я подумал слишком поздно)

## Изначальный план
- Реализовать низкоуровневую логику тензоров (от форм и шагов до автограда)
- Реализовать оптимизированные вычисления на CPU
- Реализовать оптимизированные вычисления на GPU (CUDA)
- Создать толстую высокоуровневую обёртку на Python (слои, даталоадеры, ...)


## Что сделано (за 1.5 месяца с полного нуля без серьёзного опыта в C/C++)
#### Логика тензора
Произвольные формы, шаги, отступы (shapes, strides, offset) (для удобства с ограничением на размерность).
Тип данных оставил только float32 (для краткости и удобства, ведь в C сложно сымитировать дженерики).
Непосредственные данные тензора хранятся в отдельном хранилище (Storage), которым можно отдельно управлять (я для него сделал базовый пул памяти).
Тензоры могут делить одно общее хранилище (основа для view, transpose, ...).

#### Операции
Реализованы практически все базовые бинарные и унарные операции, редукции, рандом, matmul а так же несколько специфичных операций. 
Все бинарные и унарные операции реализовывались с учётом параллелизма (OpenMP) и автовекторизации (SIMD). 
То есть я проверял различные конструкции операций в Compiler Explorer, и пытался заставить компилятор их автовекторизовать (выдать vp-инструкции).
Для всех операций кроме matmul написано несколько сценариев одновременного параллелизма и векторизации в зависимости от непрерывности тензора (хотя numpy с pytorch кажись просто копируют рваные тензоры).

#### Автоград
Реализован механизм вычислительного графа, включая узлы операций, топологическая сортировка графа, обратные функции (рабочий backward_matmul - просто оргазм), управление временем жизни тензоров в графе (с учетом будущего сборщика мусора python).
Реализована поддержка нескольких графов одновременно.


## Подводные камни
### Архитектура vs Компилятор
Мне несколько раз приходилось серьёзно рефакторить архитектуру ради оптимизаций компилятора типа инлайнинга и автовекторизации. 
Так как я понимал, что написание интринсиков и asm - это даже для гиков слишком, я решил положиться на автовекторизацию. 
Но даже со всеми возможными флагами gcc/clang далеко не всегда могли правильно расплющить мои абстракции (спасибо, что хотя бы какие-то получалось).
Поэтому постоянно приходилось выворачивать наизнанку все придуманные мной диспетчеры операций и прочее.

### Сборка
Во-первых, пришлось несколько раз серьёзно повозиться с cmake для условной компиляции, проброса параметров компилятора и прочего.
Во-вторых, оказалось, что nvcc на windows требует только MSVC и точка. Никакие MSYS2 здесь не помогут. Сначала я думал, что ничего страшного, но когда попробовал потыкать MSVC в Godbolt, понял, что это вообще никуда не годится. Повезло найти выход в clang-cl, который принимает код, написанный под gcc/clang, оптимизирует, и переводит на инструкции Windows. Это казалось мне гениальным, пока я не узнал про возможности WSL.


## Что дальше?
Вообще, дальше по плану было:

- привязать python через cpp-прослойку с pybind11;

- бесконечно тестить в google colab;
- опубликовать на PyPI и настроить CI/CD для сборки колёсиков.
  
Но через 1.5 месяца работы проект пришлось зафризить из-за универа. Затем я плавно узнавал много интересного про низкоуровневую разработку. Сначала WSL - можно пойти по пути docker и забить на компиляцию под Windows. Потом я подумал, а что если оставить C для голых вычислений, а всю низкоуровневую и высокоуровневую логику тензоров на Go. Но сразу же понял, что сам Go с его сборщиком мусора и мост cgo станут bottleneck. Далее я подробнее узнал про Rust, а также открыл для себя чудесный язык Zig. После этого мне больше не охота возвращаться в чистый C/Python (я хотел повозиться с malloc/free, я повозился). Вместо этого я теперь хочу реализовать абсолютно всю логику фреймворка на Rust, голые вычисления сделать на Zig/CUDA, сделать тонкую сахарную обёртку для python и простейший порт с несколькими функциями для Go. Должно получиться что-то типо onnxruntime с возможностью обучения и инференса моделей.
